Pin the Node version and (optionally) `pnpm`, then run your normal
package manager commands. `node_modules/` already lives next to your
`package.json`, so Compendium does not add another isolation layer.
Its job is to make sure `node` and your package manager are the exact
versions you and CI agreed on.

```bash
compendium install
source <(compendium activate)
npm install            # or: pnpm install
npm run build          # or: pnpm build
```

The docs site you're reading ships its own `compendium.toml` and is
built this way. This guide walks through that setup.

## The files

<Steps>

1. **`compendium.toml`** : Node version, nothing else needed.

   ```toml
   [compendium]
   name = "compendium-docs"
   version = "0.1.0"
   min_compendium = "0.1.0"

   [registry]
   source = "public"

   [languages]
   node = "22.22.2"

   [packages]
   npm = "package.json"
   ```

   `npm` ships inside the Node tarball, so you don't pin it separately.
   Pinning Node pins npm too. The `[packages].npm` key is documentation
   today (the parser accepts it but the install path does not auto-run
   `npm install`). See [Caveats](#packagesnpm-is-parsed-but-not-yet-wired-up).

2. **`package.json`** : standard npm, nothing Compendium-specific.

   ```json
   {
     "name": "compendium-docs",
     "type": "module",
     "version": "0.0.1",
     "scripts": {
       "dev": "astro dev",
       "build": "astro build",
       "preview": "astro preview"
     },
     "dependencies": {
       "@astrojs/starlight": "^0.39.2",
       "astro": "^6.2.2",
       "sharp": "^0.34.5"
     }
   }
   ```

3. **`package-lock.json`** : generated by `npm install`, commit it.

   Locks every transitive dependency. Same lockfile + same Node = the
   same `node_modules/` on every machine.

   ```json
     }

   ```

## What Compendium sets during activate

Almost nothing in env vars, and that's the point. Node doesn't have a
`GOPATH` or a `VIRTUAL_ENV`. `node_modules/` is already per-project by
virtue of being a directory next to `package.json`. So `compendium env`
is empty for Node projects:

```
$ compendium env

Environment variables:
  NAME  ACTION  VALUE
```

The header with no body is correct: no environment variables are set
specifically for Node. What activate does do is prepend tool
directories to `PATH` so the right `node`, `npm`, and (optionally)
`pnpm` are reachable. `compendium env` does not surface PATH changes;
that is the activate script's job.

| What's on `PATH` after activate | Where it comes from                                                              |
| ------------------------------- | -------------------------------------------------------------------------------- |
| `node`, `npm`, `npx`            | `~/.local/compendium/languages/node/<ver>/bin/`. `npm` ships inside Node.        |
| `pnpm`                          | `~/.local/compendium/tools/pnpm/<ver>/bin/`. Only when `[tools].pnpm` is pinned. |
| Project binaries                | `node_modules/.bin/` (npm/pnpm puts these there; both also resolve them via `npm run`/`pnpm <script>`). |

Compendium does *not* set `NODE_PATH`, does *not* relocate the npm
cache, and does *not* manage a per-project `node_modules/`. The
standard Node resolution algorithm already gives you per-project
isolation. Compendium's only job is making sure `which node` resolves
to the version you pinned.

## Per-project isolation

```
your-app/
└── node_modules/         # already per-project, Node handles this
```

```
~/.local/compendium/
├── languages/node/22.22.2/    # shared across every project pinned to this version
└── tools/pnpm/10.33.4/        # shared across every project pinned to this version
```

Project A pinned to Node 22 and project B pinned to Node 20 each get
their own Node install (one copy per version, deduplicated across
projects). Inside each project, npm/pnpm manages `node_modules/`.

## Working with `npx` and `node_modules/.bin/`

Inside an activated project, `npx <pkg>` and `npm run <script>` both
work normally. `npx` uses the pinned `node`, and `npm run` puts
`node_modules/.bin/` on `PATH` for the duration of the script. No
Compendium-specific changes.

```bash
npm install --save-dev typescript
npx tsc --init           # uses Compendium's node, project's typescript
```

## `npm config` and the global prefix

By default, `npm install -g` puts binaries into the Node install's own
`bin/` directory, which on Compendium is
`~/.local/compendium/languages/node/<ver>/bin/`. That directory is on
`PATH` after activate, so globally installed packages do work, but
they're tied to the pinned Node version. Two consequences:

1. If you bump `[languages].node`, the new Node install starts empty.
   You'll need to reinstall any global packages.
2. Different projects pinned to different Node versions don't share
   globals.

For tools you want to standardize across the team, pin them in
`[tools]` (`pnpm`, `goreleaser`, etc.) or as a `devDependency` in
`package.json`. Both survive Node version bumps.

## Caveats

### `[packages].npm` is parsed but not yet wired up

In v0.1, only `[packages].vcpkg` triggers any install behavior. Writing:

```toml
[packages]
npm = "package.json"
```

is legal and harmless, but Compendium will not run `npm install` for you.
Do it yourself after activate. The key is reserved so future Compendium
versions can wire it up; today it serves as documentation.

### Node version in `compendium.toml` vs `engines.node` in `package.json`

Two pin points, different jobs:

| Where                              | Pins                                                          |
| ---------------------------------- | ------------------------------------------------------------- |
| `compendium.toml` → `[languages].node` | Which Node *binary* is installed on every laptop and CI runner. |
| `package.json` → `engines.node`        | The minimum Node your *package* claims to support, for downstream consumers and `npm publish` warnings. |

Pin both. Make sure `[languages].node` satisfies `engines.node`;
Compendium does not cross-check this today.

### Corepack and `packageManager` are bypassed

Recent Node ships Corepack, which reads the `packageManager` field in
`package.json` and downloads a matching pnpm/yarn on the fly. That
duplicates what Compendium is doing (pinning a package manager) and
introduces a second download path that isn't recorded in
`compendium.toml`.

Pin pnpm in `[tools]` instead, and either omit `packageManager` from
`package.json` or set it to the same version. If both disagree,
Compendium's pin wins on `PATH`, but Corepack's shim can still
intercept if it gets there first. The cleanest setup: pin in
`[tools]`, leave `packageManager` out.

Yarn and Bun aren't in the registry today. If you need them, install
them via Corepack (Yarn) or their own installer (Bun) outside
Compendium. They won't be reproducible across machines the way
`[tools].pnpm` is.

### The npm cache is global

`~/.npm` is shared across every project on your machine, regardless of
which Compendium environment is active. That's fine. The cache is
content-addressed, so different projects on different Node versions
don't conflict. If you want a fresh cache, `npm cache clean --force`
or set `npm_config_cache` in your shell.

### `node-gyp` and native modules

Packages with native code (`better-sqlite3`, `sharp`, `node-sass`)
invoke `node-gyp`, which calls the system C++ toolchain, not
Compendium's. On macOS that means Xcode Command Line Tools; on Linux
it means `build-essential`. Compendium pins Node and your package
manager, but it does not pin the C++ toolchain that `node-gyp` uses
unless you also pin `clang` or `gcc` in `[languages]` and the
package's build script honors `CC`/`CXX`. Most don't, so most native
builds still pick up whatever the host has.

If a native module build keeps failing, check whether a prebuilt
binary is available (`sharp` ships them; `node-sass` is deprecated in
favor of `sass`, which is pure JS).

## Common issues

The consolidated reference is in [Troubleshooting](/reference/troubleshooting/).
For Node projects, the most frequent reports are:

- `node: command not found` (activate wasn't sourced)
- `npm install` uses a different Node than the one pinned (a `.nvmrc`, Volta, or another shim is earlier on `PATH`)
- Corepack tries to download a different pnpm (`packageManager` in `package.json` doesn't match `[tools].pnpm`)
- Global install lands in the wrong place (`npm config get prefix` overridden by `~/.npmrc` or `NPM_CONFIG_PREFIX`)
- Native module rebuild fails after a Node version bump (ABI mismatch, run `rm -rf node_modules`)

See the [Node section](/reference/troubleshooting/#node) for fixes.
