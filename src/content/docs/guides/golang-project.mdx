Pin Go and your linters, get a project-scoped `GOPATH`, and stop arguing
about whether the bug is real or "just on my machine."

```bash
compendium install
source <(compendium activate)
go build ./...
go test ./...
```

Compendium itself ships its own `compendium.toml` and is built this way.
This guide walks through that setup.

## The files

<Steps>

1. **`compendium.toml`** : Go version and linters.

   ```toml
   [compendium]
   name = "my-go-project"
   version = "0.1.0"
   min_compendium = "0.1.0"

   [registry]
   source = "public"

   [languages]
   go = "1.26.1"

   [tools]
   golangci-lint = "2.11.4"

   [packages]
   go = "go.mod"
   ```

   Pin the Go toolchain and any tools your CI runs. Same versions on
   every laptop and every runner.

2. **`go.mod`** : standard, generated by Go itself.

   ```bash
   go mod init github.com/you/my-go-project
   ```

   Compendium does not touch `go.mod`. It is still the source of truth
   for dependencies and the minimum Go version your *code* requires.
   The version in `compendium.toml` is what gets installed; the `go`
   directive in `go.mod` is what your modules ask for.

3. **`Makefile`** (optional but recommended) : shortcuts for the common loop.

   ```make
   build:
   	go build -o bin/my-app ./cmd/my-app

   test:
   	go test -v ./...

   lint:
   	golangci-lint run
   ```

   Nothing here references Compendium. After activate, `go` and
   `golangci-lint` are on `PATH` and these targets work.

</Steps>

## What Compendium sets during activate

```
$ compendium env

Environment variables:
  NAME                    ACTION   VALUE
  GOROOT                  set      ~/.local/compendium/languages/go/1.26.1
  GOTOOLCHAIN             set      local
  GOPATH                  set      ~/.local/compendium/envs/my-go-project/go
  GOBIN                   set      ~/.local/compendium/envs/my-go-project/go/bin
```

Plus `PATH` is prepended with the managed Go binary directory and the
per-project `go/bin/`, so `go install`-ed binaries are reachable as
plain commands inside the project's shell.

| Variable      | Value                                                            | Why it matters                                                                |
| ------------- | ---------------------------------------------------------------- | ----------------------------------------------------------------------------- |
| `GOROOT`      | `~/.local/compendium/languages/go/<ver>`                         | Tells the Go tooling which `go` install to use. Pinned by `compendium.toml`.  |
| `GOTOOLCHAIN` | `local`                                                          | Prevents Go from silently downloading a different version when `go.mod` asks for one. You get the version you pinned, period. |
| `GOPATH`      | `~/.local/compendium/envs/<project>/go`                          | Per-project. `go install` and `go get` write binaries here, not `~/go/bin`.   |
| `GOBIN`       | `<GOPATH>/bin`                                                   | Where installed binaries land. Already on `PATH` after activate.              |

## Per-project `GOPATH` is the point

By default, `go install golang.org/x/tools/cmd/stringer@latest` drops a
binary into `~/go/bin/`. Two consequences:

1. Every project shares the same `stringer`, whatever version got
   installed last.
2. `~/go/bin/` accumulates binaries you stopped using years ago.

Compendium scopes `GOPATH` per project. `go install` inside a project
writes to `~/.local/compendium/envs/<project>/go/bin/`, which is on
`PATH` only while that project is activated. Project A and project B
can install incompatible versions of the same tool without stomping on
each other.

The module cache (`~/go/pkg/mod`) is *not* per-project. That's fine; the
cache is content-addressed, so multiple projects sharing it is a feature.
If you want to scope it too, set `GOMODCACHE` manually.

## `GOTOOLCHAIN=local` deserves a word

Modern Go modules can pin a toolchain version (`go 1.23.4` in `go.mod`).
By default, if you have `go 1.22.0` installed and `go.mod` says
`go 1.23.4`, the Go command downloads 1.23.4 in the background and uses
it. That defeats the point of pinning the toolchain in `compendium.toml`.

Setting `GOTOOLCHAIN=local` (which Compendium does on activate) disables
that auto-download. The toolchain you pinned is the toolchain you get.
If `go.mod` asks for something newer and the local version can't
satisfy it, the build fails loudly instead of silently switching.

If you want to bump the toolchain, edit `compendium.toml` and run
`compendium install`. One source of truth.

## Caveats

### Compendium pins the toolchain; `go.mod` declares language requirements

Two pin points, different jobs:

| Where                           | Pins                                            |
| ------------------------------- | ----------------------------------------------- |
| `compendium.toml` → `[languages].go` | Which Go *binary* is installed.           |
| `go.mod` → `go` directive       | The minimum language version your code uses.    |
| `go.mod` → `toolchain` directive | The toolchain `go.mod` would *prefer* (ignored when `GOTOOLCHAIN=local`). |

For a project where Compendium is the team's source of truth, set the
`compendium.toml` version to match what you want everyone to use, and
let `go.mod`'s `go` directive declare the language baseline. The
`toolchain` directive doesn't matter inside the activated environment.

### Lint and tool versions matter just as much as Go's version

`golangci-lint` ships a different set of analyzers per version. A CI
that runs `golangci-lint@2.11.4` and a laptop that runs whatever the
package manager felt like installing will disagree about what's broken.
Pin every tool CI touches, not just Go.

Where each tool gets pinned depends on whether the registry has it:

- **In the registry** (`golangci-lint`, `goreleaser`, and anything listed
  on [Available tools](/reference/available-tools/)): add it under
  `[tools]` in `compendium.toml`.
- **Not in the registry** (`stringer`, `mockgen`, `go-licenses`, and
  most `golang.org/x/tools/cmd/...` binaries): install them with
  `go install <pkg>@<version>` after activate. They land in the
  project's `GOBIN` and travel with the project's environment. Pin the
  version in your Makefile or CI script so every machine gets the same
  one.

### Module cache is shared, build cache is shared

`GOMODCACHE` and `GOCACHE` are not touched. Both default to locations
under `~/.cache/go-build` and `~/go/pkg/mod`. Sharing across projects
is fast and correct because both caches are content-addressed. If you
ever need to reset, `go clean -modcache` and `go clean -cache`.

### Cross-compiling still works

`GOOS=linux GOARCH=arm64 go build` works exactly as it does without
Compendium. The pinned toolchain ships every standard cross target.
Compendium does not set `GOOS` or `GOARCH`.

## Working with `go install`-ed tools

Inside an activated project, `go install example.com/foo/cmd/bar@v1.2.3`
puts `bar` on the project's `PATH` because `GOBIN` is on `PATH`. Use
this for tools that aren't in the public registry. They survive
`source <(compendium deactivate)` (the binary stays in
`envs/<project>/go/bin/`), they just leave `PATH`.

To list what's installed for this project:

```bash
ls "$GOBIN"
```

To clean it out and reinstall fresh:

```bash
rm -rf "$GOBIN"
compendium install   # restores the empty go/bin/ directory
```

## File checklist

A Compendium-ready Go project has these in the repo root:

```
my-go-project/
├── compendium.toml         # Go version + linters
├── go.mod                  # module + language baseline
├── go.sum                  # dependency checksums
├── Makefile                # optional, but useful
└── cmd/
    └── my-app/
        └── main.go
```

Clone the repo, `compendium install`, source activate, build.

## Common issues

The consolidated reference is in [Troubleshooting](/reference/troubleshooting/).
For Go projects, the most frequent reports are:

- `go: command not found` — `source <(compendium activate)` wasn't run
- `go.mod requires go >= X, but GOTOOLCHAIN=local prevents upgrade` — bump `[languages].go`
- `go install` putting binaries in `~/go/bin/` instead of the project — `GOBIN` not set or overridden
- Stale `golangci-lint` rules in CI vs laptop — pin the linter in `compendium.toml`
- `exec format error` after copying a binary across platforms — cross-compile with `GOOS`/`GOARCH`

See the [Go section](/reference/troubleshooting/#go) for fixes.
