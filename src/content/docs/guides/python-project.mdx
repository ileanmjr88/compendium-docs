Pin the Python version in `compendium.toml` and Compendium creates a
per-project venv for you. `pip` and `uv` both write into that same venv,
so you keep your existing workflow without `pyenv`, `python -m venv`, or
a `.venv/` next to the source.

```bash
compendium install
source <(compendium activate)
pip install -r requirements.txt   # or: uv sync
python main.py
```

This guide walks through the two real workflows: a pip-based project
([f1-race-replay](https://github.com/ileanmjr88/f1-race-replay)) and a
uv-based project ([safer](https://github.com/rec/safer)).

   numpy
   pyside6

   Pin versions here when you care about reproducibility (`pandas==2.2.3`).
   `pip freeze > requirements.txt` is the usual way to capture what
   actually got installed.

1. **`compendium.toml`** : Python version plus `uv` in `[tools]`.

   ```toml
   [compendium]
   name = "safer"
   version = "0.1.0"
   min_compendium = "0.1.0"

   [registry]
   source = "public"

   [languages]
   python = "3.10.20"

   [tools]
   uv = "0.10.10"
   ruff = "0.14.14"

   [packages]
   uv = "pyproject.toml"
   ```

   Pinning `uv` and `ruff` in `[tools]` means CI and every laptop get the
   same versions. uv installs itself into
   `~/.local/compendium/tools/uv/<ver>/` and lands on `PATH` after
   activate.

2. **`pyproject.toml`** : standard PEP 621 / uv project file.

   ```toml
   [project]
   name = "safer"
   version = "5.4.0"
   requires-python = ">=3.10"

   [dependency-groups]
   dev = [
       "coverage",
       "pytest",
       "ruff",
       "ty>=0.0.14",
   ]

   [tool.ruff]
   line-length = 88

   [build-system]
   requires = ["hatchling"]
   build-backend = "hatchling.build"
   ```

   Nothing here references Compendium. uv reads `pyproject.toml` as it
   always does.

3. **`uv.lock`** : generated by `uv sync`, commit it.

   Locks every transitive dependency to an exact version + hash. Same
   `uv.lock` + same Python = identical install on every machine.

</Steps>

## What Compendium sets during activate

```
$ compendium env

Environment variables:
  NAME                     ACTION    VALUE
  PYTHONHOME               unset
  VIRTUAL_ENV              set       ~/.local/compendium/envs/<project>/venv
  UV_PROJECT_ENVIRONMENT   set       ~/.local/compendium/envs/<project>/venv
  LD_LIBRARY_PATH          prepend   ~/.local/compendium/languages/python/<ver>/lib   (Linux only)
```

Plus the venv's `bin/` directory is prepended to `PATH` *before*
Compendium's language and tool directories. After activate, `python`,
`python3`, `pip`, and anything you `pip install --console-script` all
resolve to the per-project venv.

| Variable                 | Why it matters                                                                                                       |
| ------------------------ | -------------------------------------------------------------------------------------------------------------------- |
| `VIRTUAL_ENV`            | The standard signal that you're inside a venv. `pip` writes here, `python` uses it as `sys.prefix`.                  |
| `UV_PROJECT_ENVIRONMENT` | Tells uv to use Compendium's venv instead of creating a sibling `.venv/`. Without it, `uv sync` would make two venvs. |
| `PYTHONHOME`             | Forced unset. If a parent shell had it pointing at a different install, the activated Python would import from the wrong stdlib. |
| `LD_LIBRARY_PATH`        | Linux only. Points the dynamic loader at the bundled `libpython*.so` so extension modules link correctly.            |

## Per-project venv is the point

`compendium install` creates the venv at
`~/.local/compendium/envs/<project>/venv/` using the pinned Python:

```
~/.local/compendium/envs/
тФЬтФАтФА f1-race-replay/
тФВ   тФФтФАтФА venv/
тФВ       тФЬтФАтФА bin/python -> ../../../languages/python/3.13.13/bin/python3
тФВ       тФФтФАтФА lib/python3.13/site-packages/
тФФтФАтФА safer/
    тФФтФАтФА venv/
        тФФтФАтФА ...
```

The interpreter binary is shared across projects (one copy per pinned
version in `languages/python/<ver>/`); installed packages are per-project.
`f1-race-replay` on `pandas 2.2.3` and `safer` on whatever dev tools it
needs never collide.

No `.venv/` next to your source. No `pyenv local` file. No
`source .venv/bin/activate` muscle memory. `source <(compendium activate)`
is the single entry point for everything in the project.

## Venv lifecycle

The venv is rebuilt automatically when the pinned Python version changes:

```
$ # edit compendium.toml: python = "3.13.13" -> "3.14.5"
$ compendium install
тД╣  python version changed   venv recreated тАФ reinstall packages
```

Compendium reads `pyvenv.cfg` to detect the version drift and recreates
the venv against the new interpreter. Your installed packages don't
carry over. They were built against the old Python's ABI, and a fresh
install is the correct behavior. Re-run `pip install -r requirements.txt`
or `uv sync`.

If you want to wipe and start over manually:

```bash
rm -rf ~/.local/compendium/envs/<project>/venv
compendium install
```

## Working with `pip`

After activate, `pip` is the venv's pip. Standard workflow:

```bash
pip install -r requirements.txt
pip install --upgrade some-package
pip freeze > requirements.txt
```

To install a package that's also a console script (`black`, `pytest`,
`rich-cli`), it lands in the venv's `bin/` which is already on `PATH`.
You can call it directly:

```bash
pip install black
black src/
```

## Working with `uv`

If you pinned `uv` in `[tools]`, `uv sync` populates the Compendium venv
because `UV_PROJECT_ENVIRONMENT` points there:

```bash
uv sync                  # installs main + dev deps into Compendium's venv
uv sync --no-dev         # main deps only
uv add httpx             # adds to pyproject.toml + installs
uv run pytest            # runs in the same venv
```

`uv sync` does *not* create a sibling `.venv/`. If you see one appear,
either activate didn't run in this shell or something stripped
`UV_PROJECT_ENVIRONMENT`.

uv will still try to manage Python itself by default. Because
`UV_PROJECT_ENVIRONMENT` is set, it uses the existing venv's
interpreter (Compendium's) and skips its own download. You can leave
`uv python install` alone.

## Caveats

### `[packages].pip` is parsed but not yet wired up

In v0.1, only `[packages].vcpkg` triggers any install behavior. Writing:

```toml
[packages]
pip = "requirements.txt"
```

is legal and harmless, but Compendium will not run
`pip install -r requirements.txt` for you. Do it yourself after activate.
The key is reserved so future Compendium versions can wire it up; today
it serves as documentation for human readers and CI scripts.

### uv would create its own venv without `UV_PROJECT_ENVIRONMENT`

By default `uv sync` creates `.venv/` next to `pyproject.toml`. That
defeats the per-project isolation Compendium just set up. You'd end up
with two venvs and confusion about which `pytest` is running.

Compendium sets `UV_PROJECT_ENVIRONMENT` on every activate to point at
its own venv, so `uv sync` adopts it. If something in your shell or
direnv strips that variable, uv silently goes back to its default.
`compendium env` and `echo $UV_PROJECT_ENVIRONMENT` are the quick check.

### `PYTHONHOME` is force-unset, on purpose

If you ever exported `PYTHONHOME` to point at a different Python install
(some embedded SDK setups do this), it overrides the venv's `sys.prefix`
and you import the stdlib from the wrong place. Compendium unsets it on
every activate. If you need it set for a specific subprocess, set it
inline (`PYTHONHOME=... my-tool`) rather than exporting it globally.

### Python in `compendium.toml` vs `requires-python` in `pyproject.toml`

Two pin points, different jobs:

| Where                                 | Pins                                                            |
| ------------------------------------- | --------------------------------------------------------------- |
| `compendium.toml` тЖТ `[languages].python` | Which Python *binary* is installed and what the venv runs on. |
| `pyproject.toml` тЖТ `requires-python`     | The minimum Python your *code* declares as compatible.        |

`compendium.toml` is the source of truth for "what gets installed."
`requires-python` is for downstream consumers and for uv's resolver.
Pin both, and make sure `[languages].python` satisfies `requires-python`.
Compendium does not cross-check this today.

### Extensions that need system libraries

The pinned Python ships its own `libpython*.so` (Linux) or `Python.framework`
(macOS), but packages with C extensions still link against system
libraries: `pyside6` needs Qt, `psycopg2` needs `libpq`, `lxml` needs
`libxml2`. Compendium doesn't install these. Use the OS package manager,
or pick the prebuilt wheel variant if one exists
(`pyside6-essentials`, `psycopg2-binary`, `lxml-wheels`).

### The venv is at `~/.local/compendium/envs/`, not in the repo

IDE integrations (VS Code, PyCharm) that auto-detect `.venv/` in the
project root will miss this. Point them at the venv explicitly:

```
~/.local/compendium/envs/<project>/venv/bin/python
```

VS Code: "Python: Select Interpreter" тЖТ "Enter interpreter path".
PyCharm: Settings тЖТ Python Interpreter тЖТ Add тЖТ Existing environment.

### macOS does not need `LD_LIBRARY_PATH`

The bundled Python on macOS is built with `@rpath` baked in; it finds
its own libraries without help. Compendium only prepends
`LD_LIBRARY_PATH` on Linux. If you grep `compendium env` on macOS and
don't see it, that's correct.

## Common issues

The consolidated reference is in [Troubleshooting](/reference/troubleshooting/).
For Python projects, the most frequent reports are:

- `python: command not found` (activate wasn't sourced)
- `pip install` lands in the system Python's site-packages (`pip` ran before activate, or system pip is shadowing)
- `uv sync` created a `.venv/` in the repo (`UV_PROJECT_ENVIRONMENT` wasn't set, or uv is older than 0.4)
- `venv recreated тАФ reinstall packages` shows after every install (two `compendium.toml` files in scope, or Python version churn)
- `ModuleNotFoundError` for a stdlib module like `_ctypes` on Linux (`LD_LIBRARY_PATH` doesn't include the pinned Python's `lib/`)
- IDE picks the wrong interpreter (point it explicitly at the venv's `python`)

See the [Python section](/reference/troubleshooting/#python) for fixes.
